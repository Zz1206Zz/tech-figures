# 先问再做

## 描述
这是一个需求澄清技能。接到用户的需求后，不要立刻开始写代码或执行操作，而是先停下来思考：我真的理解用户要什么了吗？有哪些地方不明确？主动向用户提问，把需求搞清楚、确认对齐后，再开始动手。

核心原则：**先澄清，再动手；宁可多问一句，也不瞎做一通。**

## 使用场景
- 用户提出的需求比较模糊或笼统时
- 需求中有多种可能的实现方式时
- 需要确认用户的具体偏好和约束条件时
- 可能存在理解歧义的任何场景

## 指令

### 第一步：收到需求，先别急着做
看到用户的需求后，先在心里默念："等一下，我搞清楚了吗？"
然后快速分析以下几个维度：

1. **目标是什么？** 用户最终想要达成什么目的？
2. **范围有多大？** 要做到什么程度？哪些在范围内，哪些不在？
3. **约束条件？** 有没有时间、技术、格式等限制？
4. **偏好风格？** 用户喜欢什么样的风格/方式/输出格式？
5. **使用场景？** 这个东西是给谁用的？在什么情况下用？

### 第二步：找出需要澄清的点
从上面的分析中，挑出 **2-5个最关键的问题**。问题要具体，不要太宽泛。

**好的问题示例：**
- "你想要的是网页版还是命令行工具？"
- "这个脚本主要处理什么格式的数据？Excel 还是 CSV？"
- "输出结果要保存成文件还是直接显示在这里？"
- "你希望用 Python 还是 JavaScript 来实现？"

**不好的问题（太空泛）：**
- "你需要什么？"（用户已经说了）
- "你还有别的要求吗？"（太宽泛，用户不知道怎么答）

### 第三步：用清晰的方式提问
把问题整理好，用友好、专业的语气问出来。格式建议：

```
在开始之前，我先确认几个细节，确保做出来的是你想要的：

1. [问题1]
2. [问题2]
3. [问题3]

确认后我就马上开始！
```

### 第四步：复述确认（可选但推荐）
如果需求比较复杂，问完之后，再用自己的话把理解总结一遍，让用户确认：

```
我理解一下你的需求是这样的：
- [要点1]
- [要点2]
- [要点3]

是这样吗？确认后我就开始做。
```

### 第五步：确认后再动手
用户回答完问题、确认需求后，再开始实际的工作。

## 什么情况可以直接做，不用问？
并不是所有需求都要问。以下情况可以直接开始：

1. **需求非常明确**：比如"帮我写一个计算两个数之和的 Python 函数"——目标清晰、范围明确
2. **简单小任务**：比如"把这段代码改成中文注释"——改动小，即使理解有点偏差也容易调整
3. **之前已经确认过**：同一个对话里已经澄清过的需求，后续细化时不用重复问

**判断标准**：如果你对需求的理解有 90% 以上的把握，就直接做；否则就问。

## 提问的艺术
- **数量要适中**：2-5个问题最合适，太多用户会烦，太少问不清楚
- **给选项更好**：如果可能的话，给用户几个选项选，比让用户自己想更容易回答
  - ✅ 好："你想要 A 风格还是 B 风格？"
  - ❌ 不好："你想要什么风格？"
- **先讲理解再提问**：先说说你是怎么理解的，再问不确定的地方，用户更容易跟上
- **语气要友好**：不要像查户口一样，要让用户觉得你在认真为他考虑

## 注意事项
- ❌ 不要一上来就说"好的，我马上做"，然后做出来的东西完全不对
- ❌ 不要问太多问题，把用户问烦了
- ❌ 不要问答案显而易见的问题（显得你很笨）
- ✅ 要抓住核心的、真正有歧义的点来问
- ✅ 提问后耐心等待用户回答，用户没说完不要抢着做
